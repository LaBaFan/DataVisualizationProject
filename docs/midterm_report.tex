\begin{abstract}
FoodETA 面向外卖配送时效与延迟因素分析，基于 Kaggle 公开外卖配送数据，围绕配送时长、配送距离、天气、交通、订单时段、城市、骑手属性和车辆类型等因素，设计后续可扩展为多视图联动系统的数据分析方案。当前中期阶段已经完成数据理解、预处理脚本、距离计算工具、聚合 JSON 输出工具、分析任务定义和后续系统规划；React 前端、FastAPI 后端和交互式多视图界面属于后续工作，尚未在当前仓库中实现。
\end{abstract}

\section{项目背景}

外卖配送是即时物流中的典型场景。配送时效不仅受到配送距离影响，也会受到天气、交通密度、订单高峰、车辆类型、多单配送和骑手状态等因素共同影响。对于平台运营、配送调度和用户体验分析而言，仅观察平均配送时间不足以解释延迟订单的形成原因。

FoodETA 的目标是对外卖配送时效和延迟因素进行可视分析。项目关注的问题包括：哪些时段更容易出现延迟，天气和交通是否显著增加配送时长，短距离长耗时订单是否存在，骑手评分和车辆类型是否影响配送效率，以及不同城市或区域之间是否存在差异。当前阶段重点是建立可复现的数据处理流程和分析任务框架，为后续可视化系统实现提供稳定数据基础。

\section{数据来源与规模}

本项目使用 Kaggle 公开数据集 Food Delivery Dataset，数据集链接为：\url{https://www.kaggle.com/datasets/gauravmalik26/food-delivery-dataset}。

公开资料显示，该数据集约包含 45,000 条外卖配送记录，每条记录对应一次配送样本，字段覆盖配送时间、餐厅位置、配送位置、天气、交通、订单类型、车辆类型和骑手属性等信息。该规模满足课程建议“不少于 1 万条记录”的要求，适合用于多视图可视分析。原始数据不提交到仓库，使用者需要自行下载并放置为 \texttt{data/raw/food\_delivery.csv}。

\section{预处理流程}

当前仓库已经包含三个数据处理脚本：\texttt{scripts/preprocess.py}、\texttt{scripts/compute\_distance.py} 和 \texttt{scripts/aggregate\_data.py}。

\texttt{scripts/preprocess.py} 是主入口，运行方式如下：

\begin{verbatim}
python scripts/preprocess.py
\end{verbatim}

该脚本读取 \texttt{data/raw/food\_delivery.csv}，将原始字段统一为 snake\_case，并进一步对齐到清洗后 schema。脚本兼容常见空格、大小写差异和 NaN 字符串；从 \texttt{Time\_taken(min)} 中提取配送时长数字；当 \texttt{Time\_Orderd} 缺失时尝试使用 \texttt{Time\_Order\_picked} 解析订单小时；根据 \texttt{Order\_Date} 解析星期。

\texttt{scripts/compute\_distance.py} 实现经纬度合法性检查和 haversine 距离计算。预处理阶段会删除缺失或明显异常的配送时长、经纬度和距离记录，并生成清洗后的 \texttt{data/processed/orders\_clean.csv}。

\texttt{scripts/aggregate\_data.py} 负责生成面向后续可视化的数据摘要，输出包括 \texttt{overview\_summary.json}、\texttt{delivery\_time\_distribution.json}、\texttt{distance\_time\_sample.json}、\texttt{time\_period\_summary.json}、\texttt{hour\_summary.json}、\texttt{weather\_traffic\_summary.json}、\texttt{courier\_vehicle\_summary.json} 和 \texttt{city\_summary.json}。

脚本运行时会打印原始记录数、清洗后记录数、删除记录数、主要字段缺失比例、延迟阈值和输出文件列表。若原始 CSV 不存在，脚本会给出明确错误提示并以非零状态退出。

\section{派生变量}

预处理阶段生成以下主要派生变量：

\begin{itemize}[leftmargin=2em]
  \item \texttt{delivery\_duration\_min}：从 \texttt{Time\_taken(min)} 中提取的配送时长，单位为分钟；
  \item \texttt{distance\_km}：使用 haversine 公式计算的餐厅到配送地点距离，单位为千米；
  \item \texttt{order\_hour}：订单小时，优先来自 \texttt{Time\_Orderd}，缺失时尝试 \texttt{Time\_Order\_picked}；
  \item \texttt{weekday}：由 \texttt{Order\_Date} 解析得到的星期；
  \item \texttt{time\_period}：时段标签，包括 \texttt{breakfast}、\texttt{lunch\_peak}、\texttt{afternoon}、\texttt{dinner\_peak}、\texttt{night} 和 \texttt{unknown}；
  \item \texttt{is\_delayed}：配送时长是否严格大于全体订单配送时长的 75 分位数阈值；
  \item \texttt{speed\_kmph}：近似配送速度，异常速度置为空值。
\end{itemize}

\section{分析任务}

当前项目定义六个核心分析任务。

\begin{description}[leftmargin=2em, style=nextline]
  \item[T1 配送时间整体分布] 分析 \texttt{delivery\_duration\_min} 的中心趋势、离散程度和长尾延迟情况，判断整体配送效率是否稳定。
  \item[T2 配送距离与配送时间关系] 基于 \texttt{distance\_km} 和 \texttt{delivery\_duration\_min} 分析距离与时长的关系，识别短距离长耗时或长距离高效率的异常样本。
  \item[T3 天气与交通影响] 比较 \texttt{weather} 和 \texttt{traffic\_density} 组合下的平均配送时长、延迟率和订单量，判断外部环境压力。
  \item[T4 不同时段效率差异] 基于 \texttt{order\_hour}、\texttt{weekday} 和 \texttt{time\_period} 分析早餐、午高峰、下午、晚高峰和夜间的订单量与配送效率差异。
  \item[T5 骑手属性与车辆类型影响] 分析 \texttt{delivery\_person\_age}、\texttt{delivery\_person\_ratings}、\texttt{vehicle\_type} 和 \texttt{multiple\_deliveries} 对配送表现的影响。
  \item[T6 高延迟风险场景组合识别] 通过 \texttt{time\_period}、\texttt{weather}、\texttt{traffic\_density}、\texttt{distance\_km} 和 \texttt{multiple\_deliveries} 等字段组合，识别延迟风险较高的典型场景。
\end{description}

\section{初步可视化设计}

后续系统计划采用多视图布局。Overview 视图展示总订单数、有效订单数、平均配送时长、延迟率、平均距离、天气类别数和交通类别数。配送时间分布视图使用直方图和箱线图展示时长分布。距离-时间散点图展示 \texttt{distance\_km} 与 \texttt{delivery\_duration\_min} 的关系，并支持异常点筛选。时间视图展示小时和时段维度的订单量、平均时长和延迟率。天气交通视图使用热力矩阵或分组柱状图比较外部条件影响。骑手车辆视图按 \texttt{vehicle\_type}、评分分箱和年龄分箱比较配送表现。城市视图用于展示不同城市或区域的配送差异。

交互设计将围绕筛选、brushing、图表点击联动和详情面板展开。当前阶段尚未实现前端界面，以上内容作为后续 React 可视化系统的设计依据。

\section{经典设计借鉴与适配}

可视分析系统设计需要借鉴已有研究中成熟的表达方式和交互思想，但本项目不会直接复用现有系统，而是结合外卖配送数据重新定义分析任务、视图组合和交互流程。当前阶段主要将 LineUp、Parallel Sets 和 TimeNotes 作为后续视图设计参考。

\subsection*{LineUp：多属性排序与配送风险场景}

LineUp 的核心思想是支持多属性数据的排序、比较和权重组合分析。本项目计划借鉴这一思想，设计 Delivery Risk Ranking View，即配送延迟风险排序视图。该视图不直接罗列单条订单，而是将订单按条件组合聚合为配送风险场景，例如晚高峰 + 高交通密度 + 雨天、夜间 + 多单配送、远距离 + 高交通密度、低评分骑手 + 多单配送、节日 + 晚高峰等。

每个风险场景计划包含 \texttt{order\_count}、\texttt{avg\_delivery\_duration\_min}、\texttt{delay\_rate}、\texttt{avg\_distance\_km}、\texttt{multiple\_delivery\_rate}、\texttt{avg\_rating} 和 \texttt{risk\_score} 等指标。中期阶段暂定的综合风险评分为：

\begin{verbatim}
risk_score =
0.4 * normalized_avg_delivery_duration
+ 0.3 * delay_rate
+ 0.2 * normalized_avg_distance
+ 0.1 * multiple_delivery_rate
\end{verbatim}

该公式不是机器学习模型，而是用于可解释排序的可视分析指标，后续可以根据课程反馈调整权重。团队的设计贡献在于将 LineUp 的多属性排序思想适配到外卖配送场景，把天气、交通密度、时段、距离、多单配送和骑手评分组合成配送延迟风险场景，用于识别高延迟风险条件组合。

\subsection*{Parallel Sets：多类别路径与延迟因素组合}

Parallel Sets 适合展示多个类别变量之间的流向关系和组合分布。本项目计划借鉴这一思想，设计 Delay Factor Flow View，即配送延迟因素流向图。该视图可展示 \texttt{weather -> traffic\_density -> time\_period -> is\_delayed} 等路径，也可以扩展为 \texttt{city -> traffic\_density -> time\_period -> is\_delayed} 或 \texttt{vehicle\_type -> multiple\_deliveries -> is\_delayed}。

路径宽度表示订单数量，颜色表示延迟率或平均配送时长。该视图用于回答哪些天气和交通组合更容易导向延迟、晚高峰是否放大高交通密度影响、多单配送是否更容易与延迟共同出现，以及哪些订单量不大但延迟率较高的条件组合值得关注。团队的设计贡献在于将 Parallel Sets 的类别路径表达适配为外卖配送条件组合分析，相比单独柱状图更强调多因素组合对配送时效的影响。

\subsection*{TimeNotes：时间序列注释与配送时段解释}

TimeNotes 强调在时间序列中标注重要事件、阶段或异常点，使用户能够结合上下文理解时间模式。本项目计划借鉴这一思想，增强 Temporal Pattern View，形成 Annotated Temporal Pattern View。

该视图将在小时趋势中标注 breakfast、lunch\_peak、afternoon、dinner\_peak 和 night 等业务时段，并标注订单量最高的小时、平均配送时长最高的小时、延迟率最高的小时以及异常峰值时段。示例注释包括“12:00 午高峰，订单量达到全天峰值”“19:00 晚高峰，平均配送时长较高”“22:00 夜间订单减少，但延迟率仍偏高”。团队的设计贡献在于把 TimeNotes 的时间注释思想适配到外卖配送时段分析，使时间趋势图不仅显示数值变化，也解释高峰和异常出现的业务语境。

总体而言，本项目自己的贡献不是简单更换数据集，而是围绕“发现问题、定位异常、解释原因、排序比较”的分析流程，将配送时间、距离、天气、交通、时段、骑手和车辆因素组织到统一的延迟风险分析框架中。

\section{后续计划}

后续将基于已经生成的清洗数据和聚合 JSON 设计初步图表截图或静态样例，并继续检查距离、时段和延迟阈值是否符合分析预期。随后计划使用 React + TypeScript + Vite 实现前端界面，使用 ECharts 和 D3 构建主要图表，并实现多视图联动、筛选器、异常点选择和详情面板。
