\section{项目背景}

外卖配送是即时物流中的典型场景。配送时效不仅受到配送距离影响，也会受到天气、交通密度、订单高峰、车辆类型、多单配送和骑手状态等因素共同影响。对于平台运营、配送调度和用户体验分析而言，仅观察平均配送时间不足以解释延迟订单的形成原因。

FoodETA 的目标是对外卖配送时效和延迟因素进行可视分析。项目关注的问题包括：哪些时段更容易出现延迟，天气和交通是否显著增加配送时长，短距离长耗时订单是否存在，骑手评分和车辆类型是否影响配送效率，以及不同城市或区域之间是否存在差异。当前阶段重点是建立可复现的数据处理流程和分析任务框架，为后续可视化系统实现提供稳定数据基础。

\section{数据说明}

\subsection{数据来源与规模}

本项目使用 Kaggle 公开数据集 Food Delivery Dataset，数据集链接为：\url{https://www.kaggle.com/datasets/gauravmalik26/food-delivery-dataset}。

该数据集约包含 45,000 条外卖配送记录，每条记录对应一次配送样本，字段覆盖配送时间、餐厅位置、配送位置、天气、交通、订单类型、车辆类型和骑手属性等信息。

\subsection{预处理流程}

当前仓库已经包含三个数据处理脚本：\texttt{scripts/preprocess.py}、\texttt{scripts/compute\_distance.py} 和 \texttt{scripts/aggregate\_data.py}。

\texttt{scripts/preprocess.py} 是主入口，运行方式如下：

\begin{verbatim}
python3 scripts/preprocess.py
\end{verbatim}

该脚本读取 \texttt{data/raw/food\_delivery.csv}，将原始字段统一为 snake\_case，并进一步对齐到清洗后 schema。从 \texttt{Time\_taken(min)} 中提取配送时长数字；当 \texttt{Time\_Orderd} 缺失时尝试使用 \texttt{Time\_Order\_picked} 解析订单小时；根据 \texttt{Order\_Date} 解析星期。

\texttt{scripts/compute\_distance.py} 实现经纬度合法性检查和 haversine 距离计算。预处理阶段会删除缺失或明显异常的配送时长、经纬度和距离记录，并生成清洗后的 \texttt{data/processed/orders\_clean.csv}。

\texttt{scripts/aggregate\_data.py} 负责生成面向后续可视化的数据摘要，输出包括 \texttt{overview\_summary.json}、\texttt{delivery\_time\_distribution.json}、\texttt{distance\_time\_sample.json}、\texttt{time\_period\_summary.json}、\texttt{hour\_summary.json}、\texttt{weather\_traffic\_summary.json}、\texttt{courier\_vehicle\_summary.json}、\texttt{city\_summary.json}、\texttt{risk\_scenario\_summary.json}、\texttt{delay\_factor\_flow.json} 和 \texttt{time\_annotations.json}。

脚本运行时会打印原始记录数、清洗后记录数、删除记录数、主要字段缺失比例、延迟阈值和输出文件列表。

\subsection{派生变量}

预处理阶段生成以下主要派生变量：

\begin{itemize}[leftmargin=2em]
  \item \texttt{delivery\_duration\_min}：从 \texttt{Time\_taken(min)} 中提取的配送时长，单位为分钟；
  \item \texttt{distance\_km}：使用 haversine 公式计算的餐厅到配送地点距离，单位为千米；
  \item \texttt{order\_hour}：订单小时，优先来自 \texttt{Time\_Orderd}，缺失时尝试 \texttt{Time\_Order\_picked}；
  \item \texttt{weekday}：由 \texttt{Order\_Date} 解析得到的星期；
  \item \texttt{time\_period}：时段标签，包括 \texttt{breakfast}、\texttt{lunch\_peak}、\texttt{afternoon}、\texttt{dinner\_peak}、\texttt{night} 和 \texttt{unknown}；
  \item \texttt{is\_delayed}：配送时长是否严格大于全体订单配送时长的 75 分位数阈值；
  \item \texttt{speed\_kmph}：近似配送速度，异常速度置为空值。
\end{itemize}

\section{分析任务定义}

当前项目定义八个核心分析任务，覆盖整体分布、距离关系、外部条件、时间模式、骑手车辆因素、相对延迟、风险场景排序和条件路径流向。

\begin{description}[leftmargin=2em, style=nextline]
  \item[T1 配送时间整体分布] 分析 \texttt{delivery\_duration\_min} 的中心趋势、离散程度和长尾延迟情况，判断整体配送效率是否稳定。
  \item[T2 配送距离与配送时间关系] 基于 \texttt{distance\_km} 和 \texttt{delivery\_duration\_min} 分析距离与时长的关系，识别短距离长耗时或长距离高效率的异常样本。
  \item[T3 天气与交通影响] 比较 \texttt{weather} 和 \texttt{traffic\_density} 组合下的平均配送时长、延迟率和订单量，判断外部环境压力。
  \item[T4 不同时段效率差异] 基于 \texttt{order\_hour}、\texttt{weekday} 和 \texttt{time\_period} 分析早餐、午高峰、下午、晚高峰和夜间的订单量与配送效率差异。
  \item[T5 骑手属性与车辆类型影响] 分析 \texttt{delivery\_person\_age}、\texttt{delivery\_person\_ratings}、\texttt{vehicle\_type} 和 \texttt{multiple\_deliveries} 对配送表现的影响。
  \item[T6 同距离条件下的相对延迟分析] 在相近配送距离条件下比较订单是否相对更慢，避免只用全局配送时长阈值造成远距离订单天然更容易被判为延迟。
  \item[T7 高延迟风险场景排序与解释] 将天气、交通密度、时段、距离区间、车辆类型、多单配送等条件组合成配送场景，构建可解释风险评分，并比较不同场景的订单量、平均配送时长、延迟率、平均距离和多单比例。
  \item[T8 配送条件路径与延迟结果流向分析] 将 \texttt{weather -> traffic\_density -> time\_period -> is\_delayed} 或 \texttt{vehicle\_type -> multiple\_deliveries -> is\_delayed} 建模为类别路径，分析不同配送条件组合最终流向延迟或非延迟结果的比例差异。
\end{description}

\section{设计方案}

系统采用纯前端静态数据架构。数据处理层由 Python、pandas 和 numpy 组成，负责从 \texttt{data/raw/food\_delivery.csv} 生成清洗后的订单数据和聚合 JSON。前端使用 React、TypeScript、Vite 和 ECharts 构建单页 dashboard，并直接读取仓库中的 \texttt{data/processed/*.json} 静态聚合数据。
